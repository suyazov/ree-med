# AI handoff: как Codex и Kimi работают над проектом

Документ нужен, чтобы несколько AI-агентов не ломали работу друг друга и не уходили в бесконечный анализ. Его можно давать любому агенту перед задачей по теме `trimed`.

## Роли

- **Постановщик задачи**: пользователь или ведущий агент. Формулирует один понятный результат, URL, Figma-фрейм и критерии готовности.
- **Исполнитель**: агент, который меняет файлы. Работает маленькими шагами и показывает `git diff`.
- **Проверяющий**: другой агент или человек. Смотрит сайт, diff, адаптив и только после этого разрешает commit/push/deploy.

Один агент не должен одновременно быть и исполнителем, и финальным проверяющим, если задача визуальная или затрагивает формы.

## Главное правило

Одна задача = один измеримый результат.

Плохая задача:
> Доделай сайт по Figma, поправь формы, проверь все страницы и сделай рефакторинг.

Хорошая задача:
> Сделай главную страницу по Figma AS2: desktop `3116:33`, mobile `3116:538`. Менять только шаблон главной, ACF-поля и CSS главной. В конце показать `git diff --stat`, список файлов и URL проверки.

## Обязательный цикл работы

1. До изменений:
   ```bash
   git status --short
   git log --oneline -3
   ```

2. Если есть чужие изменения:
   - не откатывать;
   - не форматировать соседние файлы;
   - зафиксировать в ответе, какие файлы уже были изменены;
   - работать только с файлами своей задачи.

3. Перед правками:
   - назвать файлы, которые будут изменены;
   - если задача из Figma, указать файл, frame/node desktop и mobile;
   - если нужны ассеты, сохранить их в понятную папку, например `assets/img/main/`.

4. Во время работы:
   - сначала внести изменения в файлы;
   - только потом запускать проверки;
   - не делать длинный аудит вместо реализации.

5. После изменений:
   ```bash
   git status --short
   git diff --stat
   git diff -- path/to/changed-file
   ```

6. Проверка:
   - открыть нужные URL;
   - проверить desktop и mobile;
   - проверить формы, если задача их затрагивала;
   - убедиться, что нет горизонтального скролла и явных наложений текста.

7. Завершение:
   - кратко перечислить сделанное;
   - перечислить измененные файлы;
   - написать, что не проверялось;
   - только после этого commit/push, если это было в задаче.

## Когда агент считается зависшим

Агент считается неэффективным, если после 10-15 минут работы:

- `git status --short` не показывает измененных файлов по задаче;
- агент продолжает читать Figma JSON, маппить ассеты или писать отчеты;
- агент запускает новые subagents без реальных правок;
- агент повторяет уже собранный контекст.

В этом случае не писать `продолжай` в общем виде. Нужно дать жесткую команду:

```text
Стоп анализ. У тебя нет изменений в файлах. За следующие 10 минут внеси рабочие правки только в эти файлы: ...
Не запускай subagents. Не пересобирай Figma JSON. В конце покажи git status и git diff --stat.
```

Если после второй попытки изменений нет, задачу забирает другой агент.

## Правила для Kimi

Kimi часто хорошо собирает контекст, но может уходить в анализ. Поэтому ей нужно давать короткий промпт:

```text
Задача: [один результат].
Рабочая папка: /var/www/test1.sy3.ru/wp-content/themes/trimed
Менять только: [список файлов или типов файлов].
Figma: [ссылка], desktop node [id], mobile node [id].
Не делай общий аудит. Не запускай subagents без необходимости. Не печатай base64/пиксели/длинные JSON.
Сначала внеси изменения. Потом проверь. В конце покажи:
1. git status --short
2. git diff --stat
3. список измененных файлов
4. что проверено на сайте
```

Если Kimi нужна только для разведки, задача должна звучать иначе:

```text
Только собрать факты, без изменения файлов. Ответить списком: секции, размеры, цвета, ассеты, проблемные места. Не менять код.
```

## Правила для Codex

Codex отвечает за контроль результата:

- проверяет `git status` до и после работы другого агента;
- не принимает ответ агента без diff;
- останавливает сессию, если агент застрял в анализе;
- не коммитит чужие неотслеживаемые файлы без явного решения;
- если задача срочная, после одной неудачной попытки Kimi забирает реализацию на себя.

## Git-правила

- Не использовать `git reset --hard`, `git checkout -- .`, `git clean -fd` без прямого разрешения.
- Коммит должен содержать только файлы задачи.
- Перед commit:
  ```bash
  git diff --check
  git status --short
  ```
- Для WordPress-темы при изменении CSS/JS поднять `TRIMED_VERSION` в `functions.php`.
- Не коммитить `.env`, дампы, временные отчеты, скриншоты аудита и случайно скачанные ассеты, если они не нужны сайту.

## Проверка WordPress-проекта

Минимальный набор команд:

```bash
php -l functions.php
php -l page-disinfection.php
curl -I https://test1.sy3.ru/
curl -I https://test1.sy3.ru/stomatologiya/
curl -I https://test1.sy3.ru/medcentry/
curl -I https://test1.sy3.ru/laboratoriya/
```

Если менялись формы, дополнительно проверить:

- есть ли сообщение об успехе для пользователя;
- есть ли сообщение об ошибке;
- куда уходит письмо;
- не сломался AJAX endpoint `admin-ajax.php`.

## Шаблон постановки задачи исполнителю

```text
Проект: TRIMED WordPress theme.
Рабочая папка: /var/www/test1.sy3.ru/wp-content/themes/trimed

Задача: [коротко один результат]

Источник дизайна:
- Figma: [ссылка]
- Desktop node: [id]
- Mobile node: [id]

Ограничения:
- Не трогать чужие изменения.
- Не удалять существующие блоки без прямого указания.
- Не делать общий рефакторинг.
- Менять только: [файлы/секции].

Критерии готовности:
- [пункт 1]
- [пункт 2]
- [пункт 3]

Финальный ответ исполнителя:
- git status --short
- git diff --stat
- измененные файлы
- что проверено
- что осталось рискованным или не проверено
```

## Шаблон проверки результата

```text
Проверяю работу исполнителя.

1. git status до проверки:
2. diff по измененным файлам:
3. визуальная проверка desktop:
4. визуальная проверка mobile:
5. формы/кнопки:
6. найденные ошибки:
7. можно ли коммитить: да/нет
```

## Текущее правило для главной TRIMED AS2

Если задача снова касается новой главной:

- Figma: `ТРИМЕД for AS2`
- Desktop frame: `Главная 1440`, node `3116:33`
- Mobile frame: `Главная 375`, node `3116:538`
- Figma token на сервере: `/root/.figma_token`
- Не использовать старый `.env` токен темы для AS2, он может открывать другой файл или давать 404.
- Не читать PNG как base64 и не печатать большие JSON в ответ.
- Сначала сделать рабочий шаблон/стили, потом сверять визуально.
